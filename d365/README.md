## What is here

`Publish-KritD365SalesApp.ps1` — a **codegen demonstration artifact**, committed
`a07233c` (2026-07-05 08:18, wave `.5208`): *"GENERATED BY SCXCODEX (scx-coder)
via the Kritical SCX muxing/proxy stack ... Demonstrates SCXCodex building real
D365 Sales deployment automation end to end."*

Two things to know about it:

1. **It is a demo of THIS repo's code-generation capability**, not a
   production deployment tool. It was AST-parse-validated (0 errors) but has
   never been executed against a live BC environment.
2. **The name is misleading**: despite "D365SalesApp", the function publishes
   a `.app` file to **Business Central** (BcContainerHelper, falling back to
   the BC Admin API). It is BC deploy automation, not D365 Sales.

## Where the real tooling lives

| Need | Canonical home |
|---|---|
| Publish AL apps to BC (sandbox/prod) | `KRTPax8ToShopifyConnector` — al-dev MCP `al_publish` + `Step-PaxAppPatch`/`Save-PaxWave` toolkit flows (CLAUDE.md "Auth + publish") |
| D365 **Sales** deployment automation | `KRTPax8ToShopifyConnector/scripts/d365sales/` (17-script toolchain, `Start-KritD365SalesLive.ps1`) |
| Shared PS deployment utilities | `Krit.PowerShellToolkit` |

## Verdict

- **KEEP HERE as a demo artifact** — it is evidence of the SCXCodex build
  chain working end to end, which is this repo's subject matter.
- **Do NOT wire it into BC deploy flows** without a review pass: the
  established publish path (al-dev MCP + toolkit) is tested and this is not.
- If SCXCodex generates more BC/D365 artifacts, put them under a
  `demos/`-style path with this same provenance header so folder names never
  again imply a live subproject.
